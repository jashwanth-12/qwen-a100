GPU Roofline & Quantization

A mental model for LLM inference on A100 vs H100

The Roofline Model

Every operation lives on this curve. Where you land determines what to optimize.

Key Insight
At EVERY batch size, quantized weights give you ≥ the same throughput. Faster when memory-bound, equal when compute-bound. You never lose.
The worry "A100 shouldn't use INT8 because it'll become compute-bound too fast" is backwards. Hitting compute-bound earlier means you got free performance in the memory-bound regime with no penalty after.

αhbm = C / Whbm — This single number tells you where the regime boundary is. H100 has ~4x higher ridge point than A100, so it stays memory-bound for much larger workloads.

Bottom Line
H100 is faster in both regimes because both its bandwidth AND compute are higher. The ridge point only matters for deciding which optimization strategy to pursue.

Architecture × Accelerator Matching

Pick your optimization strategy based on your hardware's ridge point.

High αhbm (H100, future chips)

Maximize work per byte loaded: large batch inference, MoE (fewer weights per token), aggressive weight quantization
Mixtral on H100 with INT4

Low αhbm (A100, older GPUs)

Don't waste compute: dense models are fine since you hit compute-bound easily; focus on batching
LLaMA-70B on A100 with bf16, large batch

Very high bandwidth (HBM3e, some TPUs)

Dense models decode fast; less pressure to quantize
Bandwidth-rich chips handle weights easily

Limited memory capacity (single GPU)

Smaller models or aggressive quantization to fit in memory at all
7B INT4 on consumer GPU

Quantization: The WxAy Framework

W = weight bits, A = activation bits. Which to reduce depends on your bottleneck.

Your Claim Verdict Why
"Lower W bits → reach compute-bound faster" Correct Fewer bytes from HBM → Bcrit drops → memory-bound region shrinks
"If both W and A are lower, less calc, lower TTFT" Correct Lower precision matmul = more TOPS on same hardware = faster compute
"W precision > A precision doesn't make sense" Correct BF16 weights × INT8 activations is pointless — weights are the big thing to shrink, and tensor cores need matching precision
"For TTFT, reduce both W and A together" Correct Both must be low to use faster tensor cores (INT8×INT8, FP8×FP8)
"If memory issues, reduce only W" Correct W4A16, W8A16 = smaller footprint in HBM, same compute speed

Important Nuance: Same FLOPs, Faster Execution

Lower precision doesn't reduce the number of operations — same matrix dimensions, same multiplications. What changes is how many FLOPs the hardware can execute per second:

BF16 tensor cores312 TFLOPS (A100)
INT8 tensor cores624 TOPS (A100) — 2x throughput
FP8 tensor cores3,958 TFLOPS (H100) — only on newer chips

Same work, faster execution — not less work.

The Complete Mental Model

WxAy
x = weight bits    y = activation bits
x < 16  →  saves MEMORY (smaller model, faster decode)
x = y < 16  →  saves COMPUTE (faster tensor cores, lower TTFT)
x ≤ y always  →  never quantize activations more than weights
x = y = hardware's best low precision  →  maximum efficiency
TL;DR
Shrink W to save memory. Shrink W+A together to save compute. Never shrink A more than W.
